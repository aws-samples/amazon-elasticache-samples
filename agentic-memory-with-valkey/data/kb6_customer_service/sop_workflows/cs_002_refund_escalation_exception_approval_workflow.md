# SOP: Refund Escalation and Exception Approval Workflow

## Purpose
Standardize refund escalation handling when a refund cannot be completed in the normal workflow or when a customer requests an exception.

## Common Scenarios
- Return outside policy window requesting exception
- Partial refund request due to minor damage or delay
- Duplicate charge concern pending payment verification
- Marketplace seller dispute requiring review
- Refund processing delay beyond standard timeline

## Baseline Workflow
1. Confirm order details and issue type
2. Verify applicable policy and any active temporary exceptions
3. Check whether system issue/incident is affecting refund visibility or processing
4. Determine if request can be resolved within agent authority
5. If not, submit escalation with required documentation

## Required Escalation Data
- Order ID
- Item(s) involved
- Refund request type (full/partial/exception)
- Applicable policy cited
- Customer impact summary
- Prior actions taken
- Evidence (photos/tracking/chat transcript excerpt if relevant)

## Agent Guidance
- Acknowledge customer concern and explain next step clearly
- Do not promise approval outcome
- Use approved policy denial / review wording when needed
- If active incident affects refund status display, disclose that status updates may be delayed

## Exception Approval Notes
Exception approvals must be based on documented policy exception paths, manager authority, or incident-related handling guidance. Agents should not create ad hoc policy exceptions.
